# NATS 演示与集群评估环境 (Docker Compose)

本目录提供了一个完整的、一键式运行的本地多节点评估环境。该环境包含了一个三节点的主 NATS 集群、一个连接到该集群的 NATS 叶子节点（Leaf Node）、当前项目的 `gnats` 监控 Web 控制台，以及一个自动化流量注入器。

---

## 架构与拓扑逻辑

本环境使用 Docker Compose 进行编排，包含以下几个服务：

```
                    ┌──────────────┐
                    │    gnats     │  (监控 UI, 运行在 8085 端口)
                    └──────┬───────┘
                           │ (可切换连接至 nats-1 或 nats-leaf)
                           ▼
 ┌──────────────────────────────────────────────────────────────┐
 │                     NATS 主集群 (Hub)                         │
 │  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐      │
 │  │    nats-1    │◄─►│    nats-2    │◄─►│    nats-3    │      │
 │  │ (Seed, JS开) │   │ (JS开启)     │   │ (JS开启)     │      │
 │  └──────┬───────┘   └──────────────┘   └──────────────┘      │
 │         ▲                                                    │
 └─────────┼────────────────────────────────────────────────────┘
           │ (叶子节点桥接 TCP 7422)
           ▼
 ┌─────────────────┐
 │    nats-leaf    │ (叶子节点, 仅开启 Core NATS, 映射主机 4223 端口)
 └─────────────────┘
           ▲
           │ (Core 消息与微服务路由透明互通)
           ▼
 ┌─────────────────┐
 │  demo-injector  │ (往 nats-1 持续注入 Stream/KV/Service 模拟流量)
 └─────────────────┘
```

### 1. 三节点 NATS 主集群 (Hub)
- 服务为 `nats-1`、`nats-2` 与 `nats-3`。
- 节点之间通过 `6222` 端口建立 Mesh 路由互联，共享同一个集群名称 `"nats-cluster"`。
- 三个节点均开启了 **JetStream** (`-js`)，构成了一个高可用的 Raft 元数据与消息存储组。
- `nats-1` 作为集群的种子节点（Seed），并且额外在 `7422` 端口上启用了 `leafnodes` 监听，供外部的叶子节点进行桥接连接。

### 2. 轻量级叶子节点 (Leaf Node)
- 服务为 `nats-leaf`。通过应用 `nats-leaf.conf` 配置，将其桥接到主集群的 `nats-1:7422`。
- **关于叶子节点的 JetStream 设计**：
  - 本环境中的叶子节点**未开启本地 JetStream 存储**。
  - 这是由于 NATS 的机制设计，如果要在叶子节点上使用 JS，它必须维护本地的存储状态（Store-and-Forward 模式）。若无本地离线持久化的硬性需求，关闭本地 JS 可以极大节省边缘节点的内存与磁盘资源。
  - 尽管关闭了本地 JS，由于 Leaf Node 优秀的透明桥接特性，叶子节点依然可以通过普通的 Core NATS 协议与主集群进行无缝的消息路由。这意味着：**主集群上运行的微服务（Services）和发布的主题消息会自动同步到叶子节点，供连在叶子节点上的客户端透明地订阅和调用**。

### 3. 数据注入器 (demo-injector)
- 服务为 `demo-injector`，运行项目中的 `demo` 示例客户端。
- 在容器启动后，它会等待 `nats-1` 的 `4222` 端口通畅，并且**额外等待 8 秒**（以给 NATS 主集群完成 JetStream 选举和元数据同步提供足够的时间缓冲），随后开始连接并注入流量。
- 注入的数据包含：
  - 创建名为 `settings_kv` 的 Key-Value 桶。
  - 创建名为 `configs_object` 的对象存储桶。
  - 创建名为 `ORDERS` 的消息流（Stream）。
  - 注册并运行 `order-service` 和 `payment-service` 两个微服务（Services），并开始高频模拟发送订单创建、查询以及交易支付的请求响应流。

### 4. 监控控制台 (gnats)
- 运行当前项目的 `gnats` 服务（控制台 UI 后端），对外暴露主机的 **`8085`** 端口（为防止与本机 `8080` 常用开发端口冲突而做了调整）。
- 通过挂载 `connections.json` 预配置了两个连接选项，供用户在界面上一键点击对比：
  - **`Main NATS Cluster (nats-1)`**：直接连接至主集群。
  - **`Leaf Node (nats-leaf)`**：连接至轻量级叶子节点。

## 快速开始

### 1. 启动环境
在 `example/` 目录下（或在项目根目录下通过指定 `-f` 参数）运行以下命令：

```bash
# 在 example/ 目录下启动
docker compose up -d

# 或在项目根目录下启动
docker compose -f example/docker-compose.yml up -d
```

该命令会自动构建项目的前后端镜像，并拉起 5 个容器实例。

### 2. 访问控制台
启动成功后，在浏览器中打开：
👉 **[http://localhost:8085](http://localhost:8085)**

### 3. 对比与验证
进入控制台后，你会看到预配的两个连接：

#### 链接 A：Main NATS Cluster (nats-1)
- 连上后，你可以查看到完整的元数据面板：
  - **Stream** 页面：能看到 `ORDERS` 数据流和不断增长的消息数。
  - **KV / Object Store** 页面：可以看到被注入的 `settings_kv` 桶以及被创建的键值。
  - **Services** 页面：能看到 `order-service` 和 `payment-service` 正在接收吞吐。

#### 链接 B：Leaf Node (nats-leaf)
- 切换到该连接：
  - **KV / Object Store / Stream** 页面：显示为空（因为叶子节点没有开启本地 JS，实现了存储隔离，这完全符合轻量化叶子的预期）。
  - **Services** 页面：**依然能完整看到并调用 `order-service` 和 `payment-service`**！

### 4. 关闭清理
当你完成测试后，可以通过以下命令停止并清理所有服务和容器：

```bash
docker compose down
```
